Configuration Architecture · Design Report

One form, one JSON, five drawings.

How a stack of hardcoded, brand-specific CAD scripts became a single configuration-driven engine — where a customer fills one form and the system draws flooring, speaker, sprinkler, partition and electrical layouts from the same shared description of their store.

Schema v3.1 DXF · ezdxf · shapely 5 dockets Shared zones · $ref

01 — The problem

Every brand had its own script. None of them generalised.

Each layout type started life as a standalone Python script written for one client, with layer names, block names, labels, sizes and placement rules baked into the code — the electrical generator alone carried roughly forty hardcoded constants and a dozen block-name lists. Onboarding a new store meant editing source.

The goal of this work is to decouple the data from the logic: the customer describes their drawing once, in plain layer and block names, through a form. That produces a single JSON config. The engines keep the clever geometry but read every name, label and zone from the config instead of from constants.

Design rule throughout: the config says what (which layers, which zones, which labels) — the engine owns how (the placement maths). Anything a customer can't reasonably know stays in the engine as a default.

02 — System architecture

Form → config → pipeline → drawings

The live app (app.py serving flooring_app.html) is the front door. A customer uploads a DXF; the app reads its layers, blocks and text labels and offers them back as autocomplete. The form assembles one JSON config, which the pipeline turns into geometry and hands to each enabled docket engine.

03 — The core idea

Zones are defined once and referenced everywhere

A retail floor (FOH), a toilet, a pickup table, a clinic — these regions are needed by more than one docket. Re-listing them per docket would bloat the config and let copies drift. Instead every region lives once in a shared zones catalog, and each docket points at it with a JSON reference: {"$ref":"#/zones/<name>"}. Sibling keys alongside a $ref override it.

Defined once

"zones": {
  "Toilet":     { "identifier":"balloon",  "keyword":"TOILET" },
  "FOH":        { "identifier":"polyline", "layer":"FOH_ZONE_OUTLINE" },
  "PickupTable": { "identifier":"balloon",  "keyword":"PICKUP" },
  "Storage":    { "identifier":"polyline", "layer":"STORAGE_AREA_OUTLINE" }
}

Referenced from any docket

// flooring
{ "zone": {"$ref":"#/zones/Toilet"}, "flooring_type":"NoFloor" }

// speaker
{ "target": {"$ref":"#/zones/PickupTable"}, "position":"center" }

// electrical
{ "reference_zone": {"$ref":"#/zones/FOH"} }

A zone knows how it is found in the DXF — a closed polyline on a layer, a named block, or a balloon (a text keyword inflated to the surrounding walls). That single definition is enough for the pipeline to extract a real polygon, whichever docket asked for it.

04 — The dockets

Five layouts, one config spine

Each docket is an optional block in the config and a collapsible step in the form. Flooring is the spine; the other four are toggled on as needed. All of them draw on the same store outline and the same shared zones.

Flooring

Tile & hatch the floor

Zones are assigned a flooring type (a tile size + rotation, a hatch pattern, or skip). With no assignments the whole store is floored with the default type.

"flooring": {
  "store_outline": { "layers": ["PLANO-CARPET"] },
  "zones": [ { "zone": {"$ref":"#/zones/Toilet"}, "flooring_type":"NoFloor" } ],
  "wall_frame": { "mirror_layers": ["A-WALL", "LK-PARTITION"] }
}
Speaker

Place speakers by coverage

Each speaker covers a circle (its coverage radius — the "balloon"). Placements pick a target (a zone or the whole store) and a position; a blank count auto-fits by coverage. Amplifier boxes are a separate fixture at a zone such as the pickup table.

"speaker": {
  "model": { "coverage_radius_mm":4000, "box_width_mm":400, "box_height_mm":200 },
  "placements": [ { "target": {"$ref":"#/zones/FOH"}, "position":"corners", "count":null } ],
  "boxes":      [ { "target": {"$ref":"#/zones/PickupTable"}, "position":"center" } ]
}
Sprinkler

Full-coverage head grid

Fills the store (minus excluded zones) so every point lies within a head's radius. Spacing is min(radius·√2, max_spacing); the head count is computed, never entered.

"sprinkler": {
  "model": { "coverage_radius_mm":2300, "max_spacing_mm":3700, "pattern":"square" },
  "target": "store",
  "exclude": [ {"$ref":"#/zones/Toilet"} ]
}
Partition

Clean civil marking plan

Purges everything not on the keep-layers (a block survives only if it contains a structural layer), restyles zone outlines by colour + lineweight, and optionally labels each zone's area. Styling targets a shared zone — the zone already carries its layer and polygon.

"partition": {
  "keep": { "boundary":[…], "structural":["A-WALL"], "partitions":[…], "text":[…] },
  "style": [ { "zone": {"$ref":"#/zones/Storage"}, "color_aci":1, "lineweight":50, "label_area":true } ],
  "label_layer": "SYSTEM_LABELS"
}
Electrical

Power & data points

The richest docket. Each point group matches fixtures by layer / block name, chooses where the point goes (place selects the engine algorithm), and carries the spec label + note. The reference zone sets the outward leader direction; columns and glass layers feed ceiling-point placement.

"electrical": {
  "reference_zone": {"$ref":"#/zones/FOH"},
  "columns_layer": "column",
  "glass_layers": ["Glass-FrontGlazing", "A-GLAZING"],
  "groups": [
    { "match":["JJ_FIXTURE_","MIRROR_"], "place":"wall",    "label":"WP1", "note":"6A RAW POINT @2250MM" },
    { "match":["SCREEN_55"],            "place":"ceiling", "label":"CP4", "note":"6/16A + CAT6" },
    { "match":["QMS_DESK","BILLING_COUNTER"], "place":"floor", "label":"FP2", "note":"6/16A RAW IN FLOOR" },
    { "match":["PICKUP_TABLE"],         "place":"data",    "label":"CT1", "note":"3-CONDUIT FLOOR BOX" }
  ]
}

05 — Generalisation in practice

What the config carries vs. what the engine keeps

The electrical reference script is the clearest case. Reading it line by line, every input fell into one of two buckets — the customer's facts, which became config, and the drawing intelligence, which stayed in code.

Config decides — what

  • Which layers are zones, columns, glass, boundary
  • Which block / layer names are which fixtures
  • The point category & placement intent (place)
  • Every visible label and spec note

Engine decides — how

  • Snap-to-wall, project-to-room-wall, façade clustering
  • The AR fixture split by distance to the FOH boundary
  • Switchboard walk that avoids curtains; column avoidance
  • Symbol sizes, leader geometry, dimensioning, colours (defaults)

This split is also why the electrical form landed on point groups rather than four bare lists: the category isn't just a name bucket, it selects a placement algorithm. Four fields per group — match, place, label, note — is the minimum that still draws a real, annotated electrical plan instead of unlabeled dots.

06 — The form

A builder that writes the JSON for you

The form is a nine-step builder with a live JSON preview. Uploading a DXF populates autocomplete for every layer, block and text label, so customers pick from what's actually in their drawing rather than typing blind. Shared zones are defined once in Step 3 and appear in every docket's dropdowns thereafter — the same dedup as the JSON, enforced in the UI.
